# Phase 6.3 — iOS 18 Per-Segment Metrics via AVMetrics — Design

> **Status:** finalized design (2026-06-24), ready for a TDD implementation plan. One question
> (response-header availability — see **PoC gate** below) gates the `cdnInfo` half only; it runs as
> **Task 1 of the plan** (a device PoC), not as a blocker on writing the plan. Closes the long-standing
> iOS coarse-segment limitation **for iOS 18+ only**; the access-log handler keeps running on all OS
> versions for its non-segment responsibilities (see Design §1).

## Motivation

iOS segment metrics are **aggregated**, not per-segment. Today `AVPlayerProbe` reads
`AVPlayerItemAccessLog`, whose entries are roll-ups, so two `SegmentMetric` fields cannot be filled
(documented in `AVAccessLogMappers.kt:11-16`):

- **`networkTiming`** is always `null` — no per-segment TTFB in the access log.
- **`cdnInfo`** degrades to `CdnHeaderParser.parse(emptyMap())` → all `UNKNOWN`/null — no response
  headers in the access log.

Android fills both from Media3's `LoadEventInfo` (`responseHeaders` + `NetworkTimingRegistry`), so the
overlay is asymmetric: the iOS Segments tab is honest but thin, and the Phase 6.2 **AGG** badge exists
precisely to admit this.

**This limitation is real but pre-iOS-18.** WWDC24 (session 10113) introduced **AVMetrics**: in iOS 18
`AVPlayerItem` conforms to `AVMetricEventStreamPublisher` and emits per-segment events as an async
sequence. Each HLS media-segment request now carries true per-request timing (backed by
`URLSessionTaskMetrics`), the CDN server address, and — pending PoC confirmation — the HTTP response.
That is exactly the data the two null fields need.

## Goal

On **iOS 18.0+**, populate `SegmentMetric.networkTiming` and `SegmentMetric.cdnInfo` with real
per-segment data, reaching parity with Android. Below iOS 18, keep the existing access-log path
unchanged (aggregated, `isAggregate = true`).

## Scope

In scope:
- A new iOS 18+ per-segment ingestion path in the Swift `AVPlayerProbe` driven by
  `AVMetricHLSMediaSegmentRequestEvent`.
- Mapping AVMetrics → existing common `SegmentMetric` (reusing `CdnHeaderParser` and the `NetworkTiming`
  model), with `isAggregate = false` on this path.
- Runtime `if #available(iOS 18, *)` gating; below iOS 18 the access-log handler emits segment metrics as
  today, and on all versions it continues to feed bitrate switches / dropped-frames / audio refresh
  (see Design §1 — the split is per responsibility, not a whole-path switch).

Out of scope (unchanged):
- Any change to the common `SegmentMetric` shape **except** an optional additive field for the CDN
  server address (see Design §4 — only if we decide to surface it).
- `SessionStore` API, `OverlayPresenter`, `OverlayViewState`, tabs/`ViewMode`.
- Stall/QoS diagnosis (Phase 6.2), FairPlay (6.1), SPM (6.4).
- tvOS/visionOS wiring beyond what falls out of the same `#available` check.

## Background — the AVMetrics surface (verified against Apple docs, all iOS 18.0+)

Subscription is an async sequence on `AVPlayerItem`:

```swift
if #available(iOS 18.0, *) {
    Task {
        for await event in playerItem.metrics(forType: AVMetricHLSMediaSegmentRequestEvent.self) {
            handleSegment(event)
        }
    }
}
```

**`AVMetricHLSMediaSegmentRequestEvent`** (per media segment):
- `url: URL?`, `byteRange: NSRange`, `mediaType: AVMediaType`, `segmentDuration: TimeInterval`,
  `isMapSegment: Bool`, `indexFileURL: URL`
- `mediaResourceRequestEvent: AVMetricMediaResourceRequestEvent?` ← the network metric

**`AVMetricMediaResourceRequestEvent`**:
- `url: URL?`, `byteRange: NSRange`, `wasReadFromCache: Bool`
- `serverAddress: String?` — the CDN IP (the value the old comment said "has no home in the model")
- `requestStartTime` / `responseStartTime` / `responseEndTime` / `requestEndTime: Date`
- `networkTransactionMetrics: URLSessionTaskMetrics?`
- `errorEvent: AVMetricErrorEvent?`

`URLSessionTaskMetrics.transactionMetrics: [URLSessionTaskTransactionMetrics]` gives, per transaction:
DNS / TCP / TLS / request / response timestamps (so **TTFB = `responseStartDate − requestStartDate`**),
and `response: URLResponse?` → `HTTPURLResponse.allHeaderFields` (the **response headers**, if present).

## Field mapping — how the two null fields get filled

| `SegmentMetric` field | iOS 18 AVMetrics source |
|---|---|
| `requestTimestampMs` | the probe's monotonic **`nowMs()`** at event-processing time — **not** `requestStartTime` (see "Clock discipline" below) |
| `totalDurationMs` | `responseEndTime − requestStartTime` (a `Date` delta — clock-agnostic) |
| `sizeBytes` | `transactionMetrics.last?.countOfResponseBodyBytesReceived`, falling back to `byteRange.length` |
| `throughputBytesPerSec` | `sizeBytes` / duration |
| `uri` | `url` |
| **`networkTiming`** | `URLSessionTaskTransactionMetrics` date breakdown (DNS/connect/TLS/**TTFB**), all `Date` deltas; `isEstimated = false` |
| **`cdnInfo`** | `transactionMetrics.last?.response → allHeaderFields` → existing `CdnHeaderParser.parse(headers)` |
| `isAggregate` | `false` (true per-segment, not a roll-up) |

The common `CdnHeaderParser` is reused verbatim — feed it the header map and provider/cache-status
detection comes for free, identical to Android.

**`sizeBytes` source:** prefer `URLSessionTaskTransactionMetrics.countOfResponseBodyBytesReceived` over
`byteRange.length`, because full-segment (non-byte-range) requests carry a zero-length `byteRange`. Fall
back to `byteRange.length` only when transaction metrics are absent.

**Clock discipline:** AVMetrics exposes wall-clock `Date`s (`requestStartTime`, `responseStartTime`,
`responseEndTime`, `requestEndTime`). The probe stamps **every** model with the monotonic `nowMs()`
(`CACurrentMediaTime()` + epoch offset) — an iOS invariant for stable DiffUtil identity and segment-list
ordering (see `sdk/src/iosMain/CLAUDE.md`). The AVMetrics path therefore uses `nowMs()` for
`requestTimestampMs` and uses the AVMetrics `Date`s **only for durations** (`totalDurationMs`, TTFB,
DNS/connect/TLS) — each a delta of two `Date`s, so unaffected by which clock domain they live in. This
keeps the segment list consistently clocked across the AVMetrics and access-log event types.

## Design

### 1. Two ingestion paths, split by responsibility (not a whole-path switch)

The Swift `AVPlayerProbe` (`Sources/StreamProbe/AVPlayerProbe.swift`) does more in its access-log handler
than emit segment metrics: `onNewAccessLogEntry` also drives **bitrate-switch detection**
(`emitBitrateSwitch` off `indicatedBitrate`), **dropped-frames errors**, and the **audio-selection
refresh** tick. `AVMetricHLSMediaSegmentRequestEvent` carries none of those. So iOS 18+ must **not**
disable the access-log path wholesale — that would silently regress variant switches, dropped-frames, and
audio refresh.

Instead the split is **per responsibility**, and the invariant is "exactly one path feeds
`addSegmentMetric`":

- The access-log observer **keeps running on all OS versions** for bitrate switches, dropped-frames, and
  audio refresh.
- `if #available(iOS 18, *)` the probe **additionally** starts an `AVMetricHLSMediaSegmentRequestEvent`
  consumer `Task`, sets an `isAVMetricsActive` flag, and routes those events to `addSegmentMetric` with
  `isAggregate = false` (per-segment).
- When `isAVMetricsActive`, the access-log handler **suppresses only its `addSegmentMetric` call** (the
  single segment-metric line in `processAccessEntry`); everything else in that handler still runs.
- Below iOS 18, `isAVMetricsActive` is false and the access-log handler emits segment metrics as today
  (aggregated, `isAggregate = true`).

The consuming `Task` is stored as a probe property and cancelled in `detach()` (and on item swap),
mirroring how the existing notification observers are torn down. It is started against the attached
`currentItem`, the same item the notification observers are scoped to.

### 2. A new iosMain mapper (parallel to `accessLogSegmentMetric`)

Add `avMetricsSegmentMetric(...)` in a **new iosMain file `AVMetricsSegmentMapper.kt`** — *not* appended
to `AVAccessLogMappers.kt` (whose KDoc is entirely about access-log degradation) and *not* in the
unrelated `AVMetricMappers.kt` (the iOS track/variant mapper, the analogue of Android's
`FormatExtensions` — nothing to do with the AVMetrics API). The function is platform-neutral in body
(it touches only `CdnHeaderParser`, `NetworkTiming`, and `SegmentMetric`, all in commonMain) but lives in
iosMain to stay parallel to `accessLogSegmentMetric`; it is therefore covered by **iosTest** (a new
`AVMetricsSegmentMapperTest`), exposed to Swift through the generated `...Kt` SKIE/ObjC bridge exactly
like `AVAccessLogMappersKt.accessLogSegmentMetric` is today.

It takes **plain Kotlin parameters** so it is unit-testable without AVFoundation types:

```kotlin
internal fun avMetricsSegmentMetric(
    requestTimestampMs: Long,
    totalDurationMs: Long,
    sizeBytes: Long,
    uri: String,
    responseHeaders: Map<String, String>,
    networkTiming: NetworkTiming?,
): SegmentMetric
```

It computes throughput, calls `CdnHeaderParser.parse(responseHeaders)`, sets `isAggregate = false`, and
attaches `networkTiming`. The Swift side does the AVMetrics→params extraction in a thin adapter and calls
this mapper through the existing SKIE bridge — mirroring how `accessLogSegmentMetric` is called today.
That adapter passes the probe's monotonic `nowMs()` as `requestTimestampMs`, computes `totalDurationMs`
from the `Date` delta, builds the header dictionary from `transactionMetrics.last?.response`, and turns
`URLSessionTaskTransactionMetrics` into a `NetworkTiming` (`isEstimated = false`) — see "Clock discipline"
above.

### 3. NetworkTiming construction

The transaction-metric dates map onto the existing `NetworkTiming` model (the same one Android fills via
`NetworkTimingRegistry`). The standard breakdown (DNS = domainLookupEnd − domainLookupStart, TCP =
connectEnd − connectStart, TLS = secureConnectionEnd − secureConnectionStart, **TTFB = responseStart −
requestStart**) is all present in `URLSessionTaskTransactionMetrics`. Because these are real per-phase
measurements (not the open()-proxy estimate), `NetworkTiming.isEstimated = false` on this path. Each
field is a `Date` delta, so the monotonic-vs-wall-clock distinction does not apply (see "Clock
discipline"). The mapping is confirmed field-by-field against the `NetworkTiming` definition during
implementation.

### 4. CDN server address (optional, additive)

`AVMetricMediaResourceRequestEvent.serverAddress` is the CDN IP the old comment lamented had "no home in
the model." Surfacing it is **optional** and out of scope unless we choose a small additive
`SegmentMetric` / `CdnHeaderInfo` field. Default: do **not** add it in this phase (YAGNI); note it as a
follow-up so the common model stays untouched and Android-identical.

### 5. Degradation within the AVMetrics path

AVMetrics is richer but not total. When a request fails at the transport layer (timeout, dropped
connection), Apple (DTS, forum #822977) confirms `networkTransactionMetrics` is `nil` and sometimes **no
segment event** is emitted (only an `errorEvent`). Handle this honestly:
- `networkTransactionMetrics == nil` → `networkTiming = null`, `cdnInfo` from whatever headers exist
  (possibly empty → `UNKNOWN`). This is *"this request didn't complete,"* **not** the old
  *"iOS can't do per-segment"* — and it is still `isAggregate = false`.
- HTTP error responses (404/500) generally **do** carry metrics, so those segments populate normally.

## PoC gate (gates `cdnInfo` only; runs as plan Task 1)

**Open question:** `URLSessionTaskTransactionMetrics.response` exists in the API since iOS 10, but Apple
does not explicitly promise AVFoundation's HLS stack populates it for media-segment transactions, and the
forum thread does not confirm header access. **`cdnInfo` parity depends entirely on this.**

**PoC (in the spirit of `AVPlayerProbePocTest` / the demo):** on a real iOS 18 device against a live
HLS stream, subscribe to `AVMetricHLSMediaSegmentRequestEvent` and log, per segment:
`mediaResourceRequestEvent?.networkTransactionMetrics?.transactionMetrics.last?.response` and its
`allHeaderFields`.

- **Headers present** → green: implement both `networkTiming` + `cdnInfo`; full Android parity.
- **Headers absent** → amber: implement `networkTiming` only (timing parity), leave `cdnInfo` as
  `UNKNOWN` on iOS even on 18+, and document the platform gap. Still a strict improvement
  (per-segment TTFB where there was none).

The `networkTiming` half does **not** depend on this gate — the timing dates are first-class properties,
not derived from `response`.

**Plan structure around the gate:** the implementation plan is a single artifact. **Task 1 is the device
PoC** (run on a real iOS 18 device; report header presence). The `networkTiming` mapper + Swift adapter +
path-split tasks proceed unconditionally. The `cdnInfo` wiring is one **explicitly conditional task**:
green → feed `CdnHeaderParser.parse(headers)`; amber → leave `cdnInfo == UNKNOWN` on iOS 18+ and document
the platform gap. The gate is a checkpoint inside the plan, not a blocker on writing it.

## Explicitly unchanged

Common `SegmentMetric` shape (modulo the consciously-deferred optional `serverAddress`), `SessionStore`
API, `OverlayPresenter`, `OverlayViewState`/`OverlayListsState`, tabs/`ViewMode`, `CdnHeaderParser`
(reused as-is), `CdnHeaderInfo`, and the Android path (byte-identical).

## Testing (TDD; hermetic — the live leg is TLS-blocked in the sandbox, as in Phase 3)

**`iosTest` (`AVMetricsSegmentMapperTest` — the part worth pinning):**
- `avMetricsSegmentMetric(...)` with a populated header map → `cdnInfo` parsed (provider/cache-status),
  `isAggregate == false`, `networkTiming` attached (`isEstimated == false`).
- `avMetricsSegmentMetric(...)` with `responseHeaders = emptyMap()` and `networkTiming = null` (failed
  transaction) → `cdnInfo == UNKNOWN`, `networkTiming == null`, still `isAggregate == false`.
- Throughput math (`sizeBytes`/duration).
- Existing `accessLogSegmentMetric(...).isAggregate == true` assertion stays green (fallback path
  intact).

The mapper body is platform-neutral but lives in iosMain (to stay parallel to `accessLogSegmentMetric`),
so its tests run under `iosSimulatorArm64Test`, not `commonTest`. The `Date`→delta extraction and
`URLSessionTaskTransactionMetrics`→`NetworkTiming` conversion are Swift-side and are exercised by the
device PoC, not the hermetic Kotlin tests.

**Manual / device PoC (gate above):** real iOS 18 device, header presence — not part of the hermetic CI
gate but the prerequisite for the `cdnInfo` work.

**Build / CI gate (full):**
`:sdk:iosSimulatorArm64Test :sdk:assembleStreamProbeCoreDebugXCFramework :sdk:assembleAndroidMain
:sdk:testAndroidHostTest :sdk:lint :sdk:ktlintCheck :sdk:detektAndroidMain :sdk:detektAndroidHostTest
:sdk:detektMetadataMain :app:assembleDebug` — plus an iOS XCFramework rebuild so the Swift AVMetrics
adapter compiles against the new mapper bridge.

## Decisions log

- **D-6.3-a** Per-segment metrics on iOS via **AVMetrics** (`AVMetricHLSMediaSegmentRequestEvent`),
  iOS 18+ only. The split is **per responsibility, not whole-path**: the access-log handler keeps running
  on all OS versions for bitrate switches / dropped-frames / audio-refresh; on iOS 18+ an `isAVMetricsActive`
  flag suppresses *only* its `addSegmentMetric` call while the AVMetrics consumer `Task` feeds segment
  metrics. "Exactly one path feeds `addSegmentMetric`" is the invariant; below iOS 18 the access-log path
  emits aggregated segment metrics as today.
- **D-6.3-b** Reuse the common `SegmentMetric` / `CdnHeaderParser` / `NetworkTiming`; add a Kotlin-param
  mapper `avMetricsSegmentMetric(...)` in a **new iosMain file `AVMetricsSegmentMapper.kt`**, parallel to
  `accessLogSegmentMetric(...)` (which is iosMain, not commonMain) and distinct from the unrelated
  `AVMetricMappers.kt` track mapper. Covered by `iosTest`. Swift does the AVMetrics→param extraction (incl.
  monotonic `nowMs()` for `requestTimestampMs` and `Date`-delta durations) in a thin adapter.
- **D-6.3-c** `isAggregate = false` on the AVMetrics path (true per-segment); iOS 18+ segments lose the
  **AGG** badge.
- **D-6.3-d** `cdnInfo` parity is **gated on a device PoC** proving `URLSessionTaskTransactionMetrics
  .response` carries headers for HLS segment transactions. `networkTiming` is ungated (first-class
  timing dates).
- **D-6.3-e** Within the AVMetrics path, a transport-layer failure (`networkTransactionMetrics == nil`)
  degrades that single segment to `networkTiming = null` / `cdnInfo = UNKNOWN` — distinct from the old
  whole-platform limitation.
- **D-6.3-f** CDN `serverAddress` surfacing deferred (optional additive field) to keep the common model
  Android-identical this phase.
